- The project is in a state where there's nothing meaningful to build and play would surface the next priority

## Decision rules (carry forward ŌĆö these matter)

1. **The system evolves from observed friction, not anticipated friction.** Ship the smallest change that makes friction visible. Watch usage. Let data drive what's next. NOT: design the full infrastructure that would solve every variant.

2. **Architectural invariants are non-negotiable** (see `VIRGIL_MASTER.md` Section 4):
   - Avrae owns mechanics; Virgil owns narrative.
   - SQLite is authoritative structured state.
   - Cloud router is the only path for LLM calls.
   - LLMs never directly mutate structural state.
   - Mode/tension/clocks/current_location_id/NPCs/locations all have single write paths.
   - Discord transport is a thin shell; no gameplay logic there.
   - Engine owns SQLite; orchestration owns rules-engine logic.
   - Avrae uses message edits as state transitions; always wire `on_message` AND `on_message_edit`.
   - `skeleton_origin=1` rows are authored canon ŌĆö parsers cannot overwrite.
   - Hard-delete commands require two independent gates (archive + phrase).

3. **The advisory parser pattern.** Five instances now: `extract_scene_updates`, Phase 3 auto-execute, Phase 11.1 mechanical hints, Phase 12A NPC extractor, Phase 12B location extractor. Pattern shape: bounded text input ŌåÆ small LLM ŌåÆ strict structured output ŌåÆ deterministic validator ŌåÆ whitelist-restricted side effect. Reuse pattern; don't proliferate architectures.

4. **One problem fixed at a time, confirmed working, then move on.** Off-roadmap fixes only when the bug is critical AND can't wait for the planned next step.

5. **Diagnostic before treatment.** When unsure: ship the smallest log line that answers the question, then decide. Session 10's two most consequential decisions (kill corpus / model swap) both came from diagnostic data, not theory. Session 13 reinforced: when a phrase comparison failed, the SQL diagnostic on the actual stored bytes (`hex(name)`) confirmed the DB was clean before any fix shipped.

6. **When user states a design preference, that's a constraint not a suggestion.** Don't make architectural calls against stated preferences. (Section 7 ŌĆö `USE_KNOWLEDGE_GUIDANCE = False` precedent.)

7. **Two independent gates before destruction** (Session 13). Any hard-delete command requires BOTH structural state (archived) AND human confirmation (typed phrase). Either gate alone is insufficient. Belt-and-suspenders explicitly applies to data loss.
